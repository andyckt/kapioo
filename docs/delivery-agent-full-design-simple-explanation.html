One-Minute Version

What we are building

A daily delivery planning assistant

The agent helps you create the best delivery plan for today's confirmed orders. It should decide which orders go into Run A, Run B, and later Run C, where drivers should meet if needed, whether support/self is required, and whether the plan can finish before 1 PM.

What it must protect

Cost and quality at the same time

The old danger was paying Google/Route Optimizer too many times while the system searched for a good plan. The new design makes the system think cheaply first, then pay for proof only after it has found a few serious plans.

The agent thinks

It uses today's orders, map coordinates, old delivery days, your business rules, and an LLM when needed.

The agent checks

Local code verifies no missing orders, no duplicate orders, good split shape, good handoff logic, and self/support only when needed.

The agent proves

Route Optimizer checks only the top finalists with Google route math. Donald approves before final routes are created.

The main rule: Route Optimizer is the final proof engine, not the search engine. The agent should not pay Google repeatedly just to discover what might work.

How the Agent Should Work Every Day

The daily flow should feel simple to you, even though the system behind it is careful.

1 Load orders Confirm today's delivery orders and ignore non-confirmed orders.
2 Check addresses Use saved coordinates and geocode cache. Warn when address quality is weak.
3 Understand the day Count orders, clusters, outliers, same-building stops, and confusing area labels.
4 Look at history Find old delivery days that looked similar and learn from good/bad outcomes.
5 Make plans Use local rules and LLM when needed to create exact Run A/B/C order splits.
6 Check plans cheaply Make sure every order is assigned once and the plan follows your rules.
7 Pick finalists Choose only a few strong, different plans worth paying to preview.
8 Pay for proof Route Optimizer previews only the finalists under a strict budget.
9 Recommend honestly Recommend only a proven qualified plan. Warn clearly if proof is partial.
10 Donald decides Choose only one of two actions: Approve, or Reject with a required reason.
From your point of view, the final product should feel like this: click one planning button, review one clear recommended plan, then either Approve it or Reject it with a reason. Final driver runs are created only after approval.

The Approve or Reject Loop

After the agent shows a recommended plan, the UI should give you only two action buttons: Approve or Reject with reason. If you reject, the reason is not just a note. It becomes the most important instruction for the next recommended plan.

1 Show one recommendation The system shows one best proven plan, plus warnings, cost used, and optional warning-only unproven ideas.
2 Approve or reject Approve creates final driver runs. Reject must include a reason, such as wrong split, meetup too late, not enough buffer, self used too much, or other notes.
3 Save the full context Admin saves the rejected plan, your reason, cost used, proof results, candidate scores, order set, and planning fingerprint.
4 LLM understands why The LLM reads your rejection reason with the rejected plan and turns it into exact planning adjustments, not vague advice.
5 Make adjustment rules The system creates hard rules, soft preferences, avoid-patterns, order moves, meetup constraints, and support/self changes for the next attempt.
6 Create next candidates The agent first tries a targeted repair. If that is not enough, the LLM generates new complete candidate plans that follow your rejection feedback.
7 Validate cheaply Local code checks every order once, no duplicates, profile rules, handoff quality, support/self logic, and deadline risk before any paid preview.
8 Preview only finalists Route Optimizer proves only the strongest revised finalists, usually fewer than the first attempt because the feedback narrowed the search.
9 Show what changed The next recommendation clearly says what changed because of your rejection reason and why this new plan is better.
10 Repeat until approved The loop repeats after each rejection, unless the system honestly cannot find a qualified plan within hard rules or cost safeguards.

What the LLM should output from your rejection

  • Reason summary: what you disliked in plain words.
  • Hard changes: things the next plan must do or must avoid.
  • Soft changes: preferences that help scoring but can be overridden by 1 PM safety.
  • Concrete adjustments: orders to move, meetup to change, buffer to improve, or self/support to adjust.
  • Risk warning: whether the feedback may make the plan impossible or more expensive to prove.

How it stays accurate

  • The LLM can reason, but local code still checks every order exactly once.
  • The next plan must be different from the rejected plan in a meaningful way.
  • The rejected reason becomes part of the planning fingerprint and cache key.
  • Route Optimizer proves the final revised plan before it can be recommended.
  • Approval stores a positive signal; rejection stores an avoid/negative signal.

Simple Example

Donald rejection reason LLM turns it into Next plan should change
"Meetup is too late. Run B cannot finish uptown after that." Hard rule: earlier meetup. Soft rule: reduce receiver burden after meetup. Move meetup earlier, let Run A do more before handoff, or use support/self only if needed.
"Self has too many orders." Hard rule: self/support should be minimized. Re-score self penalty higher. Try moving self stops back to Run A/B while still proving 1 PM finish.
"Downtown and midtown are split weirdly." Soft rule: preserve downtown/midtown grouping unless deadline proof requires otherwise. Generate a cleaner downtown/midtown run and recheck handoff/outlier impact.
"Not enough buffer before 1 PM." Hard target: improve deadline buffer. Rescue allowed if no hired-driver plan is safe. Find shorter total duration, earlier handoff, better endpoint, or minimal support/self plan.

Cost-efficient loop design

  • One LLM feedback-reasoning pass per rejection.
  • Try local targeted repair before asking for a full new plan.
  • Generate fewer candidates after rejection because your feedback narrows the search.
  • Preview only the best revised finalists, not the whole idea list.
  • Use cache when the same rejected plan and same feedback appear again.

Edge cases the loop must handle

  • Reject reason is too vague: UI asks for a clearer reason before continuing.
  • Feedback conflicts with hard rules: the system explains why it cannot follow it fully.
  • Orders change during review: old recommendation expires and the agent replans from fresh data.
  • No qualified plan exists: the agent says so instead of pretending.
  • Cost limit is near: the UI warns before more expensive rescue planning.

What gets stored after each decision

Decision Stored signal How the agent uses it later
Approve The approved plan, proof result, route shape, cost used, and current order/map fingerprint. Similar future days can trust this pattern more, especially if the actual route finishes well.
Reject with reason The rejected plan, rejection reason, LLM interpretation, avoid-pattern, and attempted adjustment rules. Similar future days should avoid the rejected pattern and prefer plans that satisfy the stored reason.
Rejected plan later fixed Before/after comparison: what changed and whether the new plan was approved. The agent learns which adjustments actually solved the problem, not just that the first plan was bad.
Final loop design: every rejection should make the next recommendation more informed. The agent should keep trying with your feedback until you approve, while still protecting hard delivery rules, cost limits, and recommendation honesty.

The Main Parts of the System

1. Kapioo Admin

This is the control center. It knows today's orders, your planning profile, your review decisions, historical learning cases, cost policy, and the final recommendation shown to you.

2. Route Optimizer

This is the route proof engine. It checks exact route sequence, time, distance, and finish time. Because it calls Google APIs, Admin should send only serious finalists to it.

3. Historical learning

This is the memory. It saves old delivery days and marks whether their route pattern was good, weak, bad, or not useful. This helps the agent avoid repeating bad patterns.

4. LLM planning

This is the flexible reasoning layer. It helps create strong candidate plans when templates are not enough, but local code and RO proof still check its work.

The LLM is not the final authority. The best design is: LLM proposes, local code validates, Route Optimizer proves, Donald approves.

How We Keep It Cheap Without Making It Weak

There are two main cost areas: Google/Route Optimizer cost and LLM token cost. The May 30/31 spike was mainly Google/Route Optimizer, not LLM. But before we add LLM, we still need cost rules so tokens do not become the next problem.

Old danger

Paying to search

The dangerous pattern is generating many plan variations and asking Route Optimizer/Google to check all of them. That can become very expensive because Google bills by stops, route legs, geocodes, and route optimization units.

New direction

Think first, prove later

The safe pattern is to use history, coordinates, local rules, and LLM reasoning to narrow the choices first. Then pay Route Optimizer to prove only the top few.

Donald's normal target

Normal daily planning should stay under $0.50/day after free credits are ignored. That is already about $15/month, so the system should still optimize lower.

Donald's rescue target

A difficult rescue day should stay under $1.00/day whenever possible. Rescue should help avoid manual work, not become a normal expensive habit.

Monthly safety limit

Monthly planning cost should warn or pause at $20/month so spending cannot quietly drift upward.

Cost control Simple meaning Quality protection
Preview budget Limit how many paid Route Optimizer previews one action can send. Choose finalists by quality signals, not just first few plans.
LLM call budget Limit how many model calls one planning session can make. Use strong model when needed; do not retry endlessly.
Cache Reuse results when today's orders and rules did not change. Same answer, lower cost, no quality loss.
Easy-day skip Do not call LLM if local logic already has a strong proven plan. Use LLM only when it adds value.
Rescue budget Keep difficult-day planning under the rescue target and stop when budget is used. Support/self can be recommended directly if it is the best proven plan.
Cost ledger Track estimated cost per planning session. You can see whether the system is worth the cost.
The goal is not "spend the least possible money." The goal is: spend a small, controlled amount only when it improves the delivery plan.

How We Protect Plan Quality

A cheap bad plan is not useful. The agent must protect business quality first: all orders delivered, before 1 PM, with the least unnecessary support.

Hard rules

  • Every confirmed order assigned exactly once.
  • No order delivered after 1 PM.
  • No fake or missing order IDs.
  • Final recommendation must have RO proof.

Business preferences

  • Prefer 5-10 minute deadline buffer.
  • Keep downtown/midtown together where sensible.
  • Keep Markham/Richmond Hill together for now.
  • Use self/support only as last resort.

Warning areas

  • Low coordinate coverage.
  • Unproven plan ideas.
  • Budget exhausted before enough proof.
  • Manual support likely required.
Exact order split
The LLM can suggest a split, but local code must verify every confirmed order ID appears once and only once. This is not optional.
Meetup quality
The system should not say "somewhere in central North York." It must select an exact meeting stop/location and prove the timing.
Self/support use
The agent should first try to make hired-driver plans work. If no safe plan can finish before 1 PM, it should recommend the best proven support/self plan directly and give you the fewest possible self/support stops.
Unproven plans
A smart-looking idea is not enough. If it has not passed local validation and Route Optimizer proof, it can be shown only as a warning or backup idea.

How Historical Learning Fits In

Historical learning is how the agent gets smarter without paying Google again. It looks at old days that already happened and turns them into useful memory.

What each old day stores

  • How many orders there were.
  • Where the orders were on the map.
  • Which stops belonged to which run.
  • Where handoff/fixed/end stops happened.
  • Whether the route finished before 1 PM.
  • Whether lateness came from route design or driver start delay.

How it helps future days

  • Find old days with similar order spread.
  • Copy useful patterns, not exact old routes blindly.
  • Avoid patterns that caused lateness.
  • Understand when a 2-driver lesson transfers to a 3-driver future.
  • Reduce paid preview calls because the search starts smarter.
Historical learning is the agent's low-cost memory. It should make the LLM prompt smaller, make candidate plans better, and reduce the need to test many plans with Route Optimizer.
Because you cannot remember exact good or bad historical dates, the system should prepare suggested labels from data first. You can then approve the first learning labels manually instead of trying to remember every old delivery day yourself.

How the LLM Should Be Used

The LLM should be used like a planner, not like an unchecked decision-maker.

Use LLM for

  • Creating complete candidate plans with exact order IDs.
  • Thinking through hard handoff and support/self tradeoffs.
  • Using historical cases and today's map shape together.
  • Explaining recommendations in plain language.
  • Understanding Donald's rejection reason and turning it into exact planning adjustments.

Do not trust LLM alone for

  • Order completeness.
  • Duplicate prevention.
  • Final finish-before-1-PM proof.
  • Paid cost budget decisions.
  • Final route creation.

Model routing in simple words

Day type Model choice Why
Easy day No LLM If local rules and cached proof already produce a strong plan, do not spend tokens.
Simple explanation or feedback parsing Cheap model These tasks are lower risk and should not need the strongest model.
Normal complex planning Strong model Exact splitting, handoff, outliers, and 1 PM deadline require stronger reasoning.
No qualified plan found Strongest rescue model Use only when the system may need self/support or manual intervention.
Agent has learned enough Cheaper model or no LLM more often As history, cache, and local rules improve, fewer days should need the strong model.
Early M22 should use a strong model by default on non-easy days. Over time, the agent should need the strong model less often because learning, cache, and local rules become smarter.

What "Live and Fully Functional" Means

The project is not truly live just because it can generate a plan. It is live when it is reliable enough for your daily business operation.

Must work

Daily planning

  • Load today's confirmed orders.
  • Generate strong candidate plans.
  • Preview finalists under budget.
  • Recommend only proven plans.
  • Offer only Approve or Reject with reason.
  • Use rejection reason to generate the next recommendation.
  • Create final driver runs after approval.
Must be visible

Donald review

  • Show latest finish time and buffer.
  • Show Run A/B/C split clearly.
  • Show handoff location and reason.
  • Show support/self reason.
  • Show why the previous plan was rejected and what changed.
  • Require a rejection reason before generating the next plan.
  • Show cost/warnings honestly.
Must be safe

Cost and fallback

  • No repeated paid previews from clicks.
  • No endless LLM retries.
  • No fake recommendation when proof is missing.
  • Manual support clearly marked.
  • Rejected-plan feedback stored and reused.
  • Cloud budgets/quotas as last protection.

Donald Decisions Recorded

These answers are now part of the current design direction.

Cost comfort

  • Normal daily planning target: under $0.50/day, still optimized as low as possible.
  • Rescue-day target: under $1.00/day, still optimized as low as possible.
  • Monthly warning/pause threshold: $20/month.

Quality behavior

  • Early M22 should use a strong model by default on non-easy days.
  • Over time, the agent should need the strong model less as it learns.
  • Unproven ideas can be visible, but must be clearly warning-only.
  • Support/self does not require special confirmation if it is the best optimized proven plan.

Historical examples

  • You cannot remember exact good/bad dates because there are too many.
  • The system should suggest learning labels from data instead of relying on memory.
  • You want to approve the first learning labels manually.

Address quality

  • Address autocomplete should happen after the Delivery Agent unless it is very small/easy.
  • Do not let address upgrade delay the Delivery Agent.
  • Use current geocode/cache approach with safeguards for now.
  • Gradually clean old addresses after the agent is working.
Remaining small decision: after we implement the agent, we can choose the exact address cleanup/autocomplete rollout plan. It should not block the delivery planning launch.

Where We Are Right Now

Already in good shape

  • Admin has route preview guardrails and cache started.
  • Historical learning case foundation exists.
  • Current deterministic candidate generation exists.
  • Donald review and final route creation flow exists.
  • The repo does not yet have LLM provider calls wired, so we can still design that layer correctly.

Final Summary

The business goal

Help you plan daily deliveries faster, finish before 1 PM, use drivers well, and use self/support only when necessary.

The design goal

Let the agent think deeply with cheap information first, then pay for Route Optimizer proof only on a few serious plans.

The launch goal

Make the agent reliable enough for daily use, with cost visibility, clear warnings, historical evaluation, and the approve/reject loop before final routes.

The safest path forward is not to rush the LLM. The safest path is to build the cost policy, fingerprint/cache, retrieval package, and evaluation harness first, then connect the LLM behind those guardrails.
The approval loop is part of the final product: if Donald rejects a plan, the agent must store the reason, improve the next plan, and use that feedback as future learning.
