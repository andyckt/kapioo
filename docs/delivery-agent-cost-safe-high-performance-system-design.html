Executive Summary

What should change

The Delivery Agent should generate and rank strong candidate plans locally, then use Route Optimizer only to prove the finalists. Cost safety should be added without making the agent cheap but weak.

  • Normal target: 30 orders today, growing slowly over time.
  • Current acceptable minimum buffer: 5 minutes before 1 PM.
  • Ideal buffer: 5-10 minutes, with current operations sometimes pushed close to 1 PM.
  • Support/self is last resort and should receive the fewest possible stops.

What should not happen

A hard low preview cap by itself can miss the best plan. The system must not simply preview the first 1-3 generated variants. It must select finalists intelligently using local quality signals.

  • Never recommend an unpreviewed candidate as proven.
  • Never hide that paid proof was budget-limited.
  • Never let repeated clicks multiply paid previews.
  • Never treat area labels as stronger than coordinates.
Final design principle: LLM, historical learning, coordinate geometry, deterministic validation, and local scoring are the search engine. Route Optimizer is the final proof engine.

Refined Final Architecture

The final architecture should be a staged planning system. Paid proof comes late, after cheap intelligence has narrowed the plan space.

1. CoordinatesPrepare lat/lng, cache hits, coverage score, and address-quality warnings.
2. FingerprintCompute today's order count, spread, clusters, outliers, and confusing area labels.
3. HistoryRetrieve similar learning cases by geography, profile compatibility, shape, and outcome quality.
4. Draft plansUse deterministic rules plus LLM to output exact order IDs per run.
5. ValidateCheck every order once, no duplicates, clusters preserved, support/self policy, profile rules.
6. Score locallyRank by deadline risk estimate, geography, driver end/homebase fit, handoff quality, history, and Donald preferences.
7. RO proofPreview only top serious finalists under budget, with exact correlation and idempotency.
8. QualifyRecommend only previewed candidates that meet 1 PM and operational trust rules.
9. ReviewDonald approves, requests improvement, or views best unproven idea as warning-only.
10. LearnStore review/outcome, build learning case, and use normalized lateness signals later.
The key design change is not "preview fewer." It is "preview fewer after selecting better finalists."

Candidate Planning Redesign

How candidate generation should evolve

  • Keep fixed templates as a fallback and baseline safety net.
  • Add LLM-generated complete candidate plans with exact order IDs per run.
  • Represent resources as Run A, Run B, Run C, not driver names.
  • Support current and future profiles: current 2 hired drivers + self backup, future 3 hired drivers, and future equal-driver mode.
  • Generate many ideas cheaply, but preview only strong finalists.

Donald's clarified operating model

  • One provider role always picks up meals from the kitchen and can bring meals north.
  • Today this is often the downtown driver, but future North York driver may become the provider.
  • Downtown/midtown route should focus downtown and may deliver midtown on the way.
  • Markham/Richmond Hill should stay together for now.
  • Self/support is not optional when no paid-driver plan can meet 1 PM, but should get the fewest possible stops.

Cost-Safe Performance Design

Donald does not yet know the manual API cost baseline and historically stayed within free credits. Therefore the system should start with conservative code budgets plus Google Cloud quotas, then show real usage so the budget can be tuned.

OptionNormal RO preview callsProsCons
Conservative 4-6 Lowest spike risk. Good while learning actual billing. May miss a better handoff/fallback plan on complex days.
Balanced recommended 8-12 Enough for 3-4 serious finalists with 2-3 runs each. Strong quality/cost balance. Still needs RO-side guardrails and cache to be fully safe.
Rescue confirmed 14-18 Useful on difficult days close to 1 PM or with unusual geography. Must require explicit confirmation and display cost risk.

Recommended starting defaults

  • Normal mode: 8-12 RO optimize-preview calls total.
  • Finalist candidates: 3-4 high-quality diverse candidates.
  • Repair/re-preview: max 2 normal repair calls.
  • Simple route preview: 1 call.
  • Rescue mode: max 14-18 calls, explicit confirmation only.

Budget exhaustion behavior

  • Stop paid preview calls immediately.
  • Return partial proof honestly.
  • Recommend only proven qualified candidates.
  • Show "best unproven idea" separately as warning-only.
  • Record correlation ID, budget used, and skipped candidates.

Geocoding Redesign

Current address capture is plain structured text. Delivery Agent has a geocode cache, but User and DailyDeliveryOrder address schemas do not currently store lat/lng/place IDs. This is a major quality and cost issue because precise meet-up and cluster decisions need coordinates.

Phase 1: fast and realistic

  • Keep using Delivery Agent geocode cache.
  • Cache successes and failures.
  • Send lat/lng to RO whenever available.
  • Warn or block recommendation when coordinate coverage is low.
  • Detect "North York" labels near kitchen by coordinate, not area string.

Phase 2: store address quality

  • Add lat/lng and normalized address fields to customer/order address records.
  • Backfill existing active customers from cache/geocode.
  • Preserve per-order override behavior because customers can change delivery address.
  • Use address hash/place ID for invalidating stale coordinates.

Phase 3: autocomplete

  • Add Google Places autocomplete to address entry/edit flows.
  • Store place ID, formatted address, lat/lng, postal code, and components.
  • Do this after cost guardrails unless it is scoped tightly.
  • Do not block Delivery Agent progress on full autocomplete migration.
Recommendation: implement cost guardrails first, then add a small address-quality milestone. Full autocomplete is valuable but touches customer flows and should be treated as a separate scoped project.

LLM Involvement

Use LLM for

  • Generating complete candidate plans with exact order IDs.
  • Reasoning over today's fingerprint and similar historical cases.
  • Explaining tradeoffs in Donald-readable language.
  • Suggesting support/self only when rules indicate hired-driver plans cannot meet 1 PM.
  • Adapting to Run A/B/C resource profiles without hardcoded driver names.

Do not trust LLM for

  • Final order completeness.
  • Duplicate prevention.
  • Hard deadline qualification.
  • Paid preview budget decisions.
  • Final route creation payload validity.

LLM output contract

  • Return max 3 complete candidate plans in normal mode.
  • Each candidate includes exact order IDs per run, planned provider/receiver roles, handoff proposal, before-handoff stops, support/self usage, and reasoning.
  • Every output is validated deterministically before any RO preview.
  • Invalid outputs are repaired locally or rejected without paid RO calls.
  • If no qualified plan is likely, LLM may say manual support is required, but deterministic proof/validation decides final status.

Recommendation and Trust Design

Qualified plan requirements

  • Every confirmed order assigned exactly once.
  • All previewed runs finish by 1 PM.
  • At least 5 minutes buffer in current operations, ideally 5-10 minutes.
  • No blocking coordinate, validation, or route-shape issue.
  • Support/self only if hired-driver plan cannot safely meet deadline.
  • RO proof exists for the recommended candidate.

When no proof is enough

  • If all previewed candidates miss 1 PM, show no safe recommendation.
  • If budget is exhausted, show best proven plan and best unproven idea separately.
  • If manual support is needed, say so directly and show the fewest support stops found.
  • If coordinate coverage is low, mark recommendation confidence low.

The UI should show the recommendation in plain operational terms: latest finish, buffer, run split, support/self use, meet-up location, why this plan, cost budget used, and warnings.

Refined Roadmap

  1. COST-1A Admin quality-preserving preview budget: finalist selector, budget object, repair cap, correlation, UI budget display.
  2. COST-1B RO paid-call budget: estimate billable units and reject high-cost previews before Google calls.
  3. COST-1C correlation/logging: unified trace across UI, Admin, RO, and Google-call estimates.
  4. COST-1D preview cache/idempotency: normalized preview payload hash and short TTL reuse.
  5. COST-1E geocode cache improvements: start/end/kitchen cache, failed geocode TTL, and customer coordinate storage plan.
  6. M20C-3E one-date learning builder: save/upsert learning cases from existing historical data only.
  7. M20C-4 backfill and uncertain match review: 30-60 days plus simple accept/edit/ignore review.
  8. M21 similarity retrieval: retrieve profile-compatible historical cases by spread, clusters, outliers, and outcomes.
  9. M22 LLM complete candidate plans: exact order IDs, run resources, support/self rules, deterministic validation.
  10. M23 budgeted RO proof loop: preview top finalists, qualify recommendation, show best unproven idea separately.
  11. M24 UI/verification polish: cost budget, confidence, tradeoff explanation, and manual support warnings.
  12. M25 future profile adaptation: 3 hired drivers, North York provider role, equal-driver mode, and role-homebase learning.
