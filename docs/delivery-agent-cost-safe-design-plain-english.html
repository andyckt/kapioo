Kapioo Delivery AI Agent Design, Explained Simply

This document explains the Delivery AI Agent in business words. It is for understanding the design before coding continues. It does not replace the technical document; it is the easier-to-read version.

1. The Simple Answer

The Delivery AI Agent should not save money by becoming weaker.

Instead, it should think more carefully before spending money. It should make smart plans using order data, map coordinates, history, rules, and LLM reasoning first. Only after that should it pay Google/Route Optimizer to check the best few plans.

Good version

The agent studies today's orders, creates strong plan ideas, picks the best few, then pays to check only those serious plans.

Bad version

The agent creates many random plan versions and pays Google to check all of them. This can become expensive very fast.

2. What Went Wrong With Cost

The expensive part is not the LLM. The expensive part is asking Route Optimizer / Google Maps to fully calculate routes too many times.

Cost spike formula:

Too many Admin preview attempts x many stops inside each route x Google billing per stop/leg/geocode = big bill.

ThingSimple meaningCost risk
Candidate plan One possible delivery split, like Run A takes these orders and Run B takes those orders. Cheap if we only create it locally.
Route Optimizer preview Ask the routing system to calculate real route time/distance. Can cost money because it uses Google APIs.
Repair / re-preview If a route looks bad, try fixing and checking it again. Can double the paid calls if uncontrolled.
Geocoding Turning text address into latitude/longitude. Can cost money if repeated often.

The old risky behavior was like asking for a full paid route calculation too early and too often.

3. The New Method: Think First, Pay Later

The agent should work more like a careful planner. It should do cheap thinking first, then pay for proof at the end.

ThinkLook at orders, addresses, areas, clusters, deadline, drivers, and history.
CreateMake several good plan ideas with exact order IDs.
FilterRemove bad plans locally before spending money.
ProvePay Route Optimizer to check only the best few serious plans.

Important rule: Route Optimizer should be used as the final checker, not as the search engine.

4. What Should Happen on a Normal Delivery Day

  1. Load today's confirmed orders. Pending or invalid orders should not be planned.
  2. Prepare addresses and coordinates. The system should know where each stop is as precisely as possible.
  3. Understand today's shape. Example: many downtown stops, some North York, Markham/Richmond Hill together, one far outlier.
  4. Look at similar old days. Find historical days with similar geography and learn what worked or failed.
  5. Create plan ideas. Plans should include exact order IDs for Run A, Run B, Run C if needed.
  6. Check the plans locally. No missing orders, no duplicates, support/self only if needed, clusters kept together where possible.
  7. Pick the serious finalists. Do not just take the first 3. Pick the strongest and most different useful options.
  8. Pay to preview finalists. Route Optimizer checks real estimated time/distance for only the finalists.
  9. Recommend only a proven plan. If no proven plan works, say so clearly.
  10. Donald approves. Only after approval should final Route Optimizer runs be created.
  11. Learn after the day. Store what happened, including whether lateness was route problem or driver-start delay.

5. How We Keep Plan Quality High

Reducing paid previews can hurt quality if done badly. The system must not simply preview fewer random plans.

Wrong way

Generate 15 plans, preview the first 3, and ignore the rest. This might miss the best plan.

Right way

Generate many ideas cheaply, score them locally, then preview the best 3-4 diverse finalists.

6. Cost Choices Explained

You said you do not yet know the normal manual Google API cost. That is reasonable. The system should start with safe options and show real usage so we can tune later.

ModePaid route previewsWhen to usePros / Cons
Very safe 4-6 calls Testing, early launch, low-risk days. Lowest cost risk, but may miss a better plan on complex days.
Balanced recommended 8-12 calls Normal daily planning. Best starting option. Enough to check 3-4 serious plans without crazy fanout.
Rescue mode 14-18 calls Hard day, close to 1 PM, unusual spread, or no safe plan found. Higher chance to find a solution, but must require your confirmation.

Recommended default: Start with Balanced mode. Show cost usage in the UI. Add Google Cloud budget/quotas so another surprise bill cannot happen even if code has a bug.

What happens if the budget runs out?

  • Stop paid previews immediately.
  • Show which plans were actually checked.
  • Only recommend a plan that was checked and passed.
  • Show the best unchecked idea separately as "unproven idea," not as the official recommendation.
  • Offer rescue mode if Donald wants to spend more preview budget that day.

7. Address and Map Accuracy

This is a big quality issue. Right now customers type addresses as text. The system does not fully use Google address autocomplete yet. That means area labels can be wrong or confusing.

Example: a customer may choose "North York," but the actual address might be close to the kitchen or better grouped with downtown/midtown. The agent should trust coordinates more than area labels.

Realistic plan

Step 1: Now

Use the existing geocode cache. Store successful and failed geocodes. Send lat/lng to Route Optimizer when available.

Step 2: Soon

Add coordinate fields to saved customer/order addresses, so good coordinates are not only temporary cache data.

Step 3: Later

Add Google address autocomplete to customer address entry/edit flows if it can be done without delaying cost guardrails too much.

Recommendation: Do not pause the whole Delivery Agent project for full autocomplete. First make cost safe. Then add a focused address-quality milestone.

The system should think in Run A, Run B, Run C, not fixed person names. You can assign real driver names yourself later.

9. Meet-Up / Handoff Design

The meet-up cannot be vague. "Somewhere in central North York" is not good enough because every minute matters.

  • The system should find an exact meet-up address or stop.
  • The provider should sometimes deliver on-the-way stops before meet-up if that saves backtracking.
  • The receiver should be able to start from the meet-up and still finish before 1 PM.
  • The meet-up should not be too close to the kitchen, too far west, too far north, or bad for the receiving driver.
  • Route Optimizer proof is important here, but only after local logic chooses good meet-up finalists.

10. What the LLM Should Do

The LLM should help make smart plans, but the system must still check its work.

LLM should do

  • Suggest complete candidate plans with exact order IDs.
  • Use history and today's map shape to reason.
  • Think flexibly about Run A, Run B, Run C.
  • Explain why a plan is good or risky.
  • Suggest support/self if it seems necessary.

LLM should not be trusted alone for

  • Making sure every order appears exactly once.
  • Final 1 PM deadline proof.
  • Paid preview budget control.
  • Final route creation.
  • Address/geocode correctness.

Best design: LLM proposes. Local code validates. Route Optimizer proves. Donald approves.

12. What Donald Should See in the UI

The UI should be simple and honest. It should not hide uncertainty.

  • Recommended plan name.
  • Run A / Run B / Run C split.
  • Latest finish time and buffer before 1 PM.
  • Whether Donald/self support is needed and why.
  • Exact meet-up location if used.
  • Cost budget used, such as "8 of 12 preview calls used."
  • Warnings if proof budget ran out.
  • Best unproven idea separately, only as warning-only.

The agent should never show an unchecked plan as the official recommendation. It may show it as an unproven idea, but the official recommendation must be proven.

13. Roadmap in Easy Words

  1. Make Admin preview safe: limit paid previews, but choose finalists intelligently.
  2. Make Route Optimizer protect itself: even if Admin has a bug, RO should reject expensive requests.
  3. Add tracking ID: every click should be traceable through Admin, RO, and Google usage.
  4. Add preview cache: if the same plan is checked twice, reuse the result instead of paying again.
  5. Improve geocoding: cache better, store coordinates, reduce repeated address conversion.
  6. Finish historical learning builder: turn old delivery days into learning cases.
  7. Backfill history: load 30-60 past days and review uncertain matches simply.
  8. Add historical similarity: use old similar days to make better plans before paid preview.
  9. Add LLM complete plan generation: LLM creates exact candidate plans with order IDs.
  10. Add final proof loop: preview only top finalists and qualify the recommendation.
  11. Polish UI: show cost, confidence, support/self reason, and warnings clearly.
  12. Support future 3-driver setup: flexible provider role, Run A/B/C resources, and future equal-driver mode.

15. Final Plain-English Summary

The Delivery AI Agent is worth building only if it can make good delivery plans. So we should not just cut cost by limiting it too hard.

The right design is: let the agent think deeply using cheap information first, then pay Google only to verify the best few plans. This keeps daily planning useful while preventing another surprise bill.

Best next coding milestone: COST-1A Admin quality-preserving preview budget. This means smart finalist selection plus paid-call limits, not blind truncation.
