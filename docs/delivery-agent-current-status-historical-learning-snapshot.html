3. Do We Have A Strong Historical Learning Database?

Yes, we now have a strong foundation. It is strong because the system has real past delivery days, real RO route information, and real coordinates.

No, it is not perfect yet. It still needs Donald review and triage so the agent knows which old days were truly good examples and which days should be treated as warning examples.

What Makes It Strong

  • It stores old Admin orders by delivery date.
  • It matches Admin orders to old Route Optimizer stops.
  • It stores which run each stop belonged to.
  • It stores route shape, timing, handoff, support/self usage, and driver/run patterns.
  • It stores coordinate spread, clusters, outliers, and area distribution.
  • It labels cases as positive, weak positive, negative, avoid pattern, uncertain, or excluded.

What Still Needs Review

  • Some dates have no RO history, so they cannot teach the agent.
  • Some dates have low Admin-to-RO matching coverage.
  • Some dates have duplicate or uncertain matches.
  • Some old routes look late or messy and should become avoid-pattern examples.
  • Donald should manually approve important learning labels later.

The database is now useful, but the safest design is not to let every historical case teach the LLM. Good cases should teach. Bad cases should warn. Unclear cases should stay out until reviewed.

4. How The LLM Uses Historical Cases

The system does not send all historical cases into the LLM. Sending everything would be expensive, confusing, and risky. Instead, the backend searches the historical database first, chooses only useful examples, summarizes them, and sends the LLM a small package.

1. Read Today Look at today’s orders, areas, coordinates, order count, and driver profile.
2. Search History Find past days that look similar to today.
3. Filter Bad Cases Remove excluded, pending, same-day, and low-quality cases.
4. Pick A Small Set Select useful positive examples, avoid examples, and context examples.
5. Send Compact Memory Send only summaries and lessons to the LLM, not raw full history.

Is This RAG?

Yes, it is RAG-like. RAG means the AI does not rely only on its own general knowledge. The system first retrieves useful information from your database, then gives that information to the LLM so it can reason with your real business history.

But this is not embedding/vector RAG yet. We are not currently turning every case into embedding vectors.

What We Use Now

  • Rule-based retrieval.
  • Similarity scoring from real delivery features.
  • Order count comparison.
  • Area distribution comparison.
  • Coordinate spread and outlier comparison.
  • Route shape and driver/resource comparison.
  • Quality gates before a case can be used.

Why This Is Good For Kapioo

  • It is cheaper than sending huge raw history to the LLM.
  • It is easier to control and inspect.
  • It avoids confusing the LLM with bad or unrelated days.
  • It is based on delivery facts, not only text similarity.

Simple version: the LLM does not read the whole library. The backend acts like a librarian. It picks a few relevant books, marks which ones are good and which ones are bad examples, then gives the LLM only the useful notes.

Should We Add Embeddings Later?

Maybe later, but not first. Embeddings are useful when you need text similarity, such as finding similar written feedback. For delivery planning, the most important similarity is geography, timing, order count, route shape, and driver resources. Those are better handled with structured logic first.

Future best design:

  • Use structured matching for route/geography similarity.
  • Use embeddings later for Donald feedback text and special-case notes.
  • Keep the LLM prompt compact so daily cost stays low.

5. Backend Logic In Simple Words

Backend Step What It Means Why It Matters
Historical backfill Builds one learning case for each old delivery day. Gives the agent memory from real business operations.
RO coordinate extraction Copies old lat/lng from Route Optimizer stops/customers. Avoids paying Google again and gives accurate map data.
Order-to-stop matching Matches Admin orders to RO route stops using order ID first, then high-confidence name/phone logic. Prevents wrong coordinates or wrong route lessons from being attached to an order.
Quality labels Marks each case as positive, weak positive, negative, avoid pattern, uncertain, or excluded. The LLM should copy good days, avoid bad patterns, and ignore garbage.
Retrieval Searches for a few relevant past cases instead of sending all cases. Keeps cost low and planning quality higher.
Compact prompt package Sends summarized lessons to the LLM, not raw customer/private data. Controls token cost and reduces privacy/noise risk.
RO final proof Uses Route Optimizer only to prove a small number of serious candidate plans. Prevents another Google API fanout spike.

6. Full Plan Until The Agent Is Fully Functional

Phase 1: Cost Safety Foundation

Status: Mostly done

  • Limit how many candidate plans can be previewed.
  • Limit how many RO proof calls can happen per action.
  • Add duplicate-click locks and budget controls.
  • Stop using RO as the search engine.

Phase 2: Historical Learning Foundation

Status: Built, needs review UI later

  • Build historical learning cases from Admin + RO.
  • Extract historical lat/lng from RO.
  • Classify cases by quality.
  • Keep bad/unclear cases from being trusted blindly.

Phase 3: Today’s Data Readiness

Status: Next important step

  • Make sure today’s orders have coordinates before the LLM plans.
  • Use Admin cache and RO/geocode cache safely.
  • Only geocode missing new addresses when necessary.
  • Show warnings for bad or unclear addresses.

Phase 4: LLM Planning Engine

Status: Partly wired, not production-ready yet

  • Use DeepSeek V4 Pro first, as Donald requested.
  • Send compact today data and compact historical memory.
  • Ask the LLM to create exact candidate split plans with order IDs.
  • Require strict JSON output so the backend can verify it.
  • Use local scoring to rank and reject nonsense before RO proof.

Phase 5: Proof, Recommendation, Approval

Status: Planned

  • Only send serious candidates to RO for final proof.
  • Recommend the best proven plan.
  • Show unproven ideas separately as warning-only options.
  • Donald gets two main actions: Approve or Reject with reason.

Phase 6: Reject Feedback Loop

Status: Planned

  • If Donald rejects, he must give a reason.
  • The LLM reads the reason and converts it into concrete changes.
  • The backend prevents repeated bad ideas.
  • The system generates the next best recommendation.
  • This repeats until Donald approves a plan.
  • The approved/rejected result becomes future learning data.

Phase 7: Production QA And Daily Use

Status: Not started yet

  • Run historical replay tests.
  • Compare LLM recommendations against known good days.
  • Measure cost per planning day.
  • Set monthly warning/pause around Donald’s $20 comfort limit.
  • Make the admin UI simple enough for daily operations.

8. Final Honest Status

Can the agent fully run the daily business yet? Not yet.

Do we now have a real learning foundation? Yes.

Is it strong enough to start serious LLM planning tests? Yes, if we use only the high-quality retrieved cases and keep weak cases out.

Will it be perfect automatically? No. We still need Donald review, today-coordinate readiness, LLM output validation, RO proof, reject feedback loop, and QA.

Are we moving in the right direction? Yes. The design now protects both quality and cost instead of sacrificing one for the other.
